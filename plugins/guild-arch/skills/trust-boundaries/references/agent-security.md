# Agent Security: Threat Catalog + Mitigations

Grounded catalog of LLM/agent-specific threats with direct quotes from primary sources: OWASP Top 10 for LLM Applications 2025, NIST AI 100-2 E2025, MITRE ATLAS, and Anthropic's published guidance. Every threat has a **mechanism**, **mitigation** (quoted), and **review question**.

## OWASP Top 10 for LLM Applications 2025 (v2.0)

Root: https://genai.owasp.org/llm-top-10/. Items below are verbatim category names with quoted mechanisms and mitigations from each item's sub-page.

### LLM01:2025 Prompt Injection

**Mechanism:**
> "A Prompt Injection Vulnerability occurs when user prompts alter the LLM's behavior or output in unintended ways."

Direct injection: "user's prompt input directly alters the behavior of the model."
Indirect injection: "an LLM accepts input from external sources, such as websites or files… alters the behavior of the model."

**Mitigations (verbatim):**
- "Provide specific instructions about the model's role, capabilities, and limitations within the system prompt"
- "Provide the application with its own API tokens for extensible functionality, and handle these functions in code rather than providing them to the model"
- "Implement human-in-the-loop controls for privileged operations"
- "Separate and clearly denote untrusted content"
- "treating the model as an untrusted user"

**Review question:** *What untrusted input does this agent read? What privilege does it have that an attacker would want?*

### LLM02:2025 Sensitive Information Disclosure

**Mechanism:**
> "Sensitive information can affect both the LLM and its application context. This includes personal identifiable information (PII), financial details, health records, confidential business data, security credentials, and legal documents."

**Mitigations (verbatim):**
- "Implement data sanitization to prevent user data from entering the training model"
- "Limit access to sensitive data based on the principle of least privilege"
- "Limit model access to external data sources"

**Review question:** *What sensitive data could this model see or infer? How is that access scoped?*

### LLM03:2025 Supply Chain

**Mechanism:**
> "LLM supply chains are susceptible to various vulnerabilities, which can affect the integrity of training data, models, and deployment platforms."

Attack scenario (verbatim): "Compromised LoRA adapters infiltrated supply chains to inject hidden vulnerabilities and malicious code."

**Mitigations (verbatim):**
- "Carefully vet data sources and suppliers"
- "Maintain an up-to-date inventory of components using a Software Bill of Materials (SBOM)"
- "Only use models from verifiable sources and use third-party model integrity checks with signing and file hashes"

**Review question:** *What models, adapters, MCP servers, or datasets is this agent using? Who signed them? Who signed the signer?*

### LLM04:2025 Data and Model Poisoning

**Mechanism:**
> "Data poisoning occurs when pre-training, fine-tuning, or embedding data is manipulated to introduce vulnerabilities, backdoors, or biases."

Attack scenario: "insert a backdoor trigger into the model… authentication bypass, data exfiltration or hidden command execution."

**Mitigations (verbatim):**
- "Track data origins and transformations using tools like OWASP CycloneDX or ML-BOM"
- "Implement strict sandboxing to limit model exposure to unverified data sources"
- "Use data version control (DVC)"

### LLM05:2025 Improper Output Handling

**Mechanism:**
> "Improper Output Handling refers specifically to insufficient validation, sanitization, and handling of the outputs generated by large language models before they are passed downstream to other components and systems."

(This is the XSS / SQLi / RCE class, where the LLM output reaches a sink that interprets it.)

**Mitigations (verbatim):**
- "Treat the model as any other user, adopting a zero-trust approach, and apply proper input validation on responses coming from the model"
- "Encode model output back to users to mitigate undesired code execution"
- "Use parameterized queries or prepared statements for all database operations involving LLM output"

**Review question:** *Where does this model's output go? Shell? SQL? HTML? A markdown renderer? Each is a sink that needs encoding.*

### LLM06:2025 Excessive Agency

**Mechanism:**
> "Excessive Agency is the vulnerability that enables damaging actions to be performed in response to unexpected, ambiguous or manipulated outputs from an LLM."

Root causes: "excessive functionality; excessive permissions; excessive autonomy."

**Mitigations (verbatim):**
- "Limit the extensions that LLM agents are allowed to call to only the minimum necessary"
- "Avoid the use of open-ended extensions where possible (e.g., run a shell command, fetch a URL, etc.)"
- "Track user authorization and security scope to ensure actions taken on behalf of a user are executed on downstream systems in the context of that specific user"
- "Utilise human-in-the-loop control to require a human to approve high-impact actions"
- "Implement authorization in downstream systems rather than relying on an LLM to decide if an action is allowed"

**Review question:** *What tools does this agent have? Which ones are open-ended (shell, arbitrary URL)? Which high-impact actions bypass a human approver?*

### LLM07:2025 System Prompt Leakage

**Mechanism:**
> "The system prompt leakage vulnerability in LLMs refers to the risk that the system prompts or instructions used to steer the behavior of the model can also contain sensitive information that was not intended to be discovered."

**Mitigations (verbatim):**
- "Avoid embedding any sensitive information (e.g. API keys, auth keys, database names, user roles, permission structure of the application) directly in the system prompts"
- "Critical controls such as privilege separation, authorization bounds checks, and similar must not be delegated to the LLM"

**Review question:** *If someone exfiltrated the full system prompt and printed it on the internet, what would break?* (If the answer is nontrivial, the system prompt is load-bearing for security. Fix that.)

### LLM08:2025 Vector and Embedding Weaknesses

**Mechanism:**
> "Vectors and embeddings vulnerabilities present significant security risks in systems utilizing Retrieval Augmented Generation (RAG) with Large Language Models (LLMs)."

Attack scenario: resume with "hidden instructions… (white text on white background)" causing an unqualified recommendation via the RAG retrieval.

**Mitigations (verbatim):**
- "Implement fine-grained access controls and permission-aware vector and embedding stores"
- "Accept data only from trusted and verified sources"
- "Tag and classify data within the knowledge base"

### LLM09:2025 Misinformation

**Mechanism:**
> "Misinformation occurs when LLMs produce false or misleading information that appears credible."

Includes hallucinations and "Overreliance… when users place excessive trust in LLM-generated content, failing to verify its accuracy."

**Mitigations (verbatim):**
- "Use Retrieval-Augmented Generation"
- "Encourage users to cross-check LLM outputs with trusted external sources"
- "Clearly communicate these risks and limitations to users"

**Review question:** *What's the failure mode if this model hallucinates? Who gets hurt? What verification does the downstream system have?*

### LLM10:2025 Unbounded Consumption

**Mechanism:**
> "Unbounded Consumption occurs when a Large Language Model (LLM) application allows users to conduct excessive and uncontrolled inferences, leading to risks such as denial of service (DoS), economic losses, model theft, and service degradation."

Sub-types: "Denial of Wallet" and "Model Extraction via API."

**Mitigations (verbatim):**
- "Implement strict input validation"
- "Apply rate limiting and user quotas"
- "Continuously monitor resource usage"

## NIST AI 100-2 E2025 (March 2025)

https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.100-2e2025.pdf — supersedes E2023.

### Jailbreaks: competing-objectives attacks

> "Prefix injection," "Refusal suppression," "Style injection," "Role-play" (e.g., DAN/AIM patterns).

### Jailbreaks: mismatched-generalization attacks

> "Special encoding" (base64), "Character transformation" (ROT13, leetspeak), "Word transformation" (payload splitting/token smuggling), "Prompt-level transformation" (translation).

### Multi-turn attacks

The **Crescendo** attack:
> "a multi-turn adaptive attack that includes seemingly benign prompts, eventually leading to a successful jailbreak."

### Indirect Prompt Injection (§3.4)

> "Because GenAI models combine the data and instruction channels, attackers can leverage the data channel to affect system operations by manipulating resources with which the system interacts."

**Attack families NIST catalogs:**
- **Availability**: time-consuming background tasks (NISTAML.017), disruptive output formatting
- **Integrity**: jailbreaking via resource; execution triggers (Neural Exec); knowledge base poisoning (PoisonedRAG, Phantom); injection hiding (non-visible content, base64)
- **Self-propagating injections (worms):**
  > "an attacker could send a malicious email that, when read by a model integrated as part of an email client, instructs the model to spread the infection by sending similar malicious emails to everyone in the user's contact list. In this way, certain malicious prompts could serve as worms."
- **Privacy**: compromising connected resources; leaking info via markdown-image exfiltration

### Security of Agents (§3.5)

> "Because agents can take actions using tools, these attacks can create additional risks in this context, such as enabling actors to hijack agents to execute arbitrary code or exfiltrate data from the environment in which they are operating."

### Mitigations and their stated limits

NIST is explicit about fragility:
> "mitigations against different classes of AML attacks tend to be empirical and limited in nature… many of these mitigations may themselves be defeated by adaptive attacks."
> "current generation models remain highly vulnerable to direct prompt injection attacks."

Suggested interventions:
- "wrapping user input in XML tags"
- "spotlighting"
- "filtering instructions from third-party data"
- "training models to follow hierarchical trust relationships in prompts"
- And critically: "application designers may design systems with the assumption that prompt injection attacks are possible if a model is exposed to untrusted input sources, such as by using multiple LLMs with different permissions or by allowing models to interact with potentially untrustworthy data sources only through well-defined interfaces."

### Information extraction (§3.3.2)

- Training-data extraction (Carlini et al. canary-style attacks)
- Prompt/context stealing: "a small set of fixed attack queries… were sufficient to extract more than 60 % of prompts"
- Model extraction of "previously unknown hidden dimensions and the embedding projection layer"

## MITRE ATLAS — Adversarial Threat Landscape for AI Systems

Root: https://atlas.mitre.org/. Agent-relevant techniques:

- **AML.T0051 LLM Prompt Injection** — "ignore aspects of its original instructions and follow the adversary's instructions instead" (direct or indirect)
- **AML.T0043 Craft Adversarial Data** — inputs engineered to elicit harmful outputs
- **AML.T0024 Exfiltration via AI Inference API** — extracting data or artifacts through queries
- **AML.T0020 Poison Training Data** — fine-tuning / RAG source manipulation
- **AML.T0086 Exfiltration via AI Agent Tool Invocation** — using agent tools to egress data
- **AML.T0110 AI Agent Tool Poisoning** — modifying tools so agent behavior is attacker-controlled

Mitigations listed against AML.T0051:
- AML.M0019 Control Access to AI Models and Data in Production
- AML.M0020 Generative AI Guardrails
- AML.M0021 Generative AI Guidelines
- AML.M0022 Generative AI Model Alignment
- AML.M0024 AI Telemetry Logging
- AML.M0033 Input and Output Validation for AI Agent Components

## Anthropic — Published Guidance

### Prompt Injection Defense

From https://www.anthropic.com/news/prompt-injection-defenses:

> "Prompt injections are adversarial instructions hidden within the content that AI models process."
> "When an agent browses the internet, it encounters content it cannot fully trust… an attacker might have embedded malicious instructions to hijack the agent."

Layered defenses as described by Anthropic:
- Training: "we expose Claude to prompt injections embedded in simulated web content, and reward it when it correctly identifies and refuses to comply"
- Classifiers: "we scan all untrusted content that enters the model's context window, and flag potential prompt injections with classifiers"
- Continuous red-teaming

Caveat (verbatim):
> "No browser agent is immune to prompt injection, and we share these findings to demonstrate progress, not to claim the problem is solved."

### Claude Code's security model

From https://code.claude.com/docs/en/security:

- "Claude Code uses strict read-only permissions by default"
- "Write access restriction… cannot modify files in parent directories without explicit permission"
- "Command blocklist: Blocks risky commands that fetch arbitrary content from the web like curl and wget by default"
- "Isolated context windows: Web fetch uses a separate context window to avoid injecting potentially malicious prompts"
- "Fail-closed matching: Unmatched commands default to requiring manual approval"
- "Trust verification: First-time codebase runs and new MCP servers require trust verification"

Best practices:
- "Avoid piping untrusted content directly to Claude"
- "Use virtual machines (VMs) to run scripts and make tool calls, especially when interacting with external web services"
- "Anthropic does not manage or audit any MCP servers" — trust is always the operator's call

## Consolidated Threat → Mitigation → Review Move

| Threat | Primary source | Mitigation (quoted / tight paraphrase) | First review move |
|---|---|---|---|
| Direct prompt injection | OWASP LLM01 | "Separate and clearly denote untrusted content"; spotlighting; "treat the model as an untrusted user" | Identify the trust boundary; mark untrusted text explicitly |
| Indirect prompt injection | NIST §3.4; OWASP LLM01 | Design "with the assumption that the AI model can and will produce malicious output"; multi-LLM pipelines with different permissions | Draw the data-flow; any external content = potential instruction source |
| Excessive agency | OWASP LLM06 | "Limit extensions to the minimum necessary"; no open-ended shell/URL; human-in-the-loop for high-impact | List every tool; for each, ask "must it exist?" and "what's its blast radius?" |
| Improper output handling | OWASP LLM05 | Zero-trust on model output; encode for each sink; parameterized queries | Follow LLM output to every sink; name the encoding strategy for each |
| System prompt leakage | OWASP LLM07 | No secrets, no authZ logic in system prompts | Read the system prompt; if it leaked, what breaks? |
| Sensitive info disclosure | OWASP LLM02 | Least-privilege data access; sanitize training/input | What data can this model reach that it shouldn't need? |
| Data/model poisoning | OWASP LLM04 | ML-BOM, DVC, sandboxed training, vendor vetting | Where does training/RAG data come from? Who signed it? |
| Vector/embedding weaknesses | OWASP LLM08 | Permission-aware vector stores; classify/tag content | Who can write to the RAG index? At what access level? |
| Misinformation / overreliance | OWASP LLM09 | RAG; cross-check; communicate limits | What verification does the downstream consumer have? |
| Unbounded consumption | OWASP LLM10 | Rate limits, quotas, resource monitoring | What's the per-request cost ceiling? What's the attacker's cost ratio? |
| Supply-chain compromise | OWASP LLM03 | Signed artifacts, SBOM, vendor vetting | List every model/adapter/MCP server; trace its provenance |
| Jailbreaks | NIST §3.3 | Safety training + guardrails + detection; red-team | Test the obvious jailbreak patterns; then test Crescendo |
| Self-propagating injections | NIST §3.4 | Hierarchical trust prompts; restrict tool effects | Can this agent send messages to other agents it impersonates? |
| Tool poisoning / tool-invocation exfil | ATLAS AML.T0110/T0086 | AML.M0033 (I/O validation), M0024 (telemetry), M0019 (access control) | Can untrusted content modify the tool or its parameters? |

## The Test That Actually Matters

From Nasr, Carlini et al., *The Attacker Moves Second* (Oct 2025):
> "By systematically tuning and scaling general optimization techniques—gradient descent, reinforcement learning, random search, and human-guided exploration—we bypass 12 recent defenses… with attack success rate above 90% for most; importantly, the majority of defenses originally reported near-zero attack success rates."

> "Notably the 'Human red-teaming setting' scored 100%, defeating all defenses."

**Implication for review:** if a mitigation's evidence is a static benchmark or a one-time red-team pass, that's not evidence. The defender has to assume adaptive attack and budget for the defense being defeated. Log enough to detect post-hoc; design enough containment to survive a defeated defense.
